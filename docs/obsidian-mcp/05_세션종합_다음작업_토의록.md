## 1. 전문가 토의 — 합의된 우선순위

| 우선순위 | 주제 | 합의 |
|----------|------|------|
| P0 | DOCX 수식 **모양(OMML)** | 깨지지 않는 것이 최우선. 평문화만으로 볼드 추구 시 이전에 **수식 붕괴** 재발. |
| P0 | LaTeX **잔재 문자** | `quad`, `Rightarrow`, `Leftrightarrow` 등이 본문에 남지 않게: `latexToPlainText` 치환 + `explanationExportGate` 경고 + 원본 md 교정 병행. |
| P1 | **문제/보기** 구간 볼드 체감 | OMML 볼드 API 한계로, `[문제]` 렌더 경로에서 **평문화 + 굵은 TextRun** 옵션(`forcePlainBoldMath`)으로 **문제 블록만** 보정. 해설 OMML은 유지. |
| P1 | 레이아웃 | 빠른정답·해설: `N. [정답] 값` 한 줄 정돈, 보기 박스 여백·긴 문항 페이지 넘김 휴리스틱 유지. |
| P2 | 해설 수식까지 “진짜 볼드 OMML” | `docx`의 `MathRun`만으로는 한계 → **OMML XML에 run 속성 주입** 등 저수준 실험은 별도 스프린트·A/B 후 채택. |

## 4. 알려진 리스크 (다음 작업에서 반복 질문 방지)

- **문제 구간** `forcePlainBoldMath`는 굵기는 확실하나, **전통 OMML “둥근” 수식 객체**와는 다르게 보일 수 있음 → 원장님 기대가 “전 구간 동일 스타일”이면 P2 실험 필요.
- 마크다운에 `,\quad`처럼 백슬래시가 빠지면 잔재로 남기 쉬움 → 게이트 경고 + 원본 수정.
- Supabase 합본 미리보기: 목록·본문 분리 로드 후에도 대형 본문+KaTeX는 무거울 수 있음.
