# 영역 지정 → 해설 → DOCX 내보내기 (베스트 프랙티스)

MCP 기반 모델 토의(Gemini·GPT)와 코드·운영 관점을 합친 **권장 동선**입니다. 목표는 한 크롭·한 문항 계약을 깨지 않고 품질 게이트를 통과시키는 것입니다.

## 1. 입력 계약 (가장 중요)

- **문제 박스(크롭) 1개 = API 호출 1회당 한 문항만.** 여러 문항이 보이면 박스를 나누거나, 시험지 레이아웃을 잘라 각 문항만 들어오게 합니다.
- **문항을 골라 풀기:** 2번·5번만 지정하고 4번은 건너뛰는 식이 운영상 흔합니다. 옆 칸·아래에 다른 번호가 비쳐도 **박스 안의 한 문항만** 생성 대상입니다.
- **해설 스타일(권장):** 수식·등호 연쇄 위주, 짧은 한국어 연결만. 프롬프트가 이 방향으로 맞춰져 있습니다.
- **선지·조건·그림이 잘리면** precheck/비전 점수가 떨어지고, 모델이 추정·환각할 확률이 올라갑니다. 테두리에 여유를 두세요.
- 텍스트 OCR 보조가 있어도 **출력 계약은 동일**: `[정답]` 한 줄 + `[해설]` 한 블록, 타 문항 인용 금지.

## 2. 파이프라인 단계 (코드와 대응)

1. **영역 지정·저장** (`page.tsx`): 페이지별 문제 박스 → 큐 → 순차 생성.
2. **추출 사전검증** (`/api/precheck-extraction`): 저품질 크롭이면 생성 전 중단.
3. **해설 생성** (`/api/generate-explanation`): Gemini 후보 순회 → 형식·정합·교과 범위·수업기준(프로필별 분량 포함) 검증 → 통과 시 선택적으로 OpenAI 교차검증(`EXPLANATION_CROSS_VERIFY`).
4. **카드 편집**: 빠른정답·해설 선택·순서 확정.
5. **DOCX** (`exportDocQuality` + 선택적 `/api/repair-explanations`): 저장 전 검증·패치. 내보내기 본문(`body`)에 **타 문항으로 이어지는 표현**·**`[정답]` 헤더 혼입** 등이 있으면 게이트에서 막힌다. repair는 JSON으로 **여러 문항을 한 번에** 정제하지만, **출력 `questionNo` 집합·개수는 입력과 동일**해야만 통과한다(병합·누락 방지).

## 3. 프롬프트·검증 원칙 (전문가 종합)

- 시스템·유저 프롬프트 모두 **단일 크롭** 전제로 통일합니다. (텍스트 다문항 파이프와 혼동하면 모델이 여러 `[정답]`을 내놓기 쉽습니다.)
- **근사·추정·장황함**은 검증 단계에서 걸러지므로, 입력 단계에서 크롭 품질을 맞추는 편이 재시도 비용을 줄입니다.
- **킬러 프로필**은 장문 허용 한도가 더 넓지만, **easy/balanced**는 글자·줄 수 상한을 더 타이트하게 두어 장황한 말풀이를 줄입니다. 그래도 한 문항 외 확장은 금지입니다.

## 4. 환경변수 (품질 우선 예시)

처음 세팅할 때는 [setup-operator-steps.md](./setup-operator-steps.md)의 순서(필수 키 → UI–env 매핑 → 교차검증)를 따르는 것을 권장합니다.  
자세한 키는 [models.md](./models.md)를 보고, 운영 시 아래를 우선 적용하세요.

- `EXPLANATION_CROSS_VERIFY=true` + `OPENAI_MODEL_CROSS_VERIFY=gpt-4o` (신뢰도)
- `OPENAI_MODEL_GENERATE_FALLBACK=gpt-4o` (Gemini 전부 실패 시)
- `OPENAI_EXPLANATION_FORMAT_RETRY=false` — **비용 절감 시에만** 끔. 기본(미설정)은 코드에서 **2차 OpenAI 호출 허용**입니다.
- Gemini 혼잡·503 대비: `GEMINI_MODELS_GENERATE_*`에 `gemini-2.0-flash` 등 여러 후보를 콤마로 나열.

## 5. 한계

- LLM·비전은 100% 정답을 보장하지 않습니다. **최종 방어선은 사람의 검수**이며, 본 앱은 실수를 줄이는 자동화 층입니다.

## 6. 해설 생성 실패(형식/정합) 시

- **단계 번호 `1. 2. 3.`** 은 정상 풀이 패턴이다. 예전 검증은 `습니다` 뒤 `2.` 줄을 타 문항 혼입으로 오탐할 수 있어, 현재 코드에서는 해당 패턴을 쓰지 않는다.
- **장문** 판정은 마침표 split 문장 수가 아니라 **글자 수·(단계형이면) 줄 수** 기준에 가깝게 맞춘다.
- **OpenAI 폴백**은 `[정답]`/`[해설]` 생략을 줄이기 위해 **system + user** 메시지로 호출한다.

- **근호 오독:** 인쇄된 `(√(2∛4))^3` 를 `√2×∛4` 의 곱으로 잘못 읽으면 답이 바뀐다. 크롭 해상도·선 굵기가 불명확하면 발생하기 쉽다. 프롬프트에서 중첩과 곱을 구분하도록 지시했으며, 교차검증(`EXPLANATION_CROSS_VERIFY`)을 켜 두면 재점검에 도움이 된다.

- **연쇄 문항 출력:** 한 응답에 `2)[정답]`, `3)[정답]`처럼 여러 문제를 붙이면 검증에서 거른다. 크롭에 문항이 하나만 보이도록 하고, 생성 실패 시 배치·다중 호출 혼선 여부를 확인한다.
