## 1. 왜 「풀이 단계」와 「해설 미리보기」에서 수식이 다르게 보일 수 있나

| 구간 | 렌더링 | 설명 |
|------|--------|------|
| 우측 패널 해설 미리보기(`renderMethodBlocks` → `renderWithMath`) | **KaTeX** (`$...$`, `$$...$$`) | API가 넣은 LaTeX 스타일 구분자가 브라우저에서 조판됨. |
| 좌측 「풀이 방법 채택」카드 요약 | 과거에는 첫 줄 **평문**만 표시 → 혼동 유발 | 현재는 요약 영역에도 **동일 `renderWithMath`** 로 짧은 구간을 조판(코드 반영). |
| DOCX 내보내기 | **평문** (√, 분수 등으로 변환) | `sanitizeExportPlainText` / `latexToPlainText` 가 `$…$`를 풀어쓴 뒤 검증·저장. 웹과 파일 형식이 의도적으로 다름. |

즉, **웹은 KaTeX**, **DOCX는 한글+유니코드 수식**이 정상 동작입니다. “파일만 예쁘다”가 아니라 **저장 형식이 다릅니다.**

## 2. 오답·이상한 전개가 나올 때 (원인 분류)

1. **근호 범위 오독:** 중첩 `√(2∛4)` vs 곱 `√2·∛4` → 답이 갈림. 크롭에 근호 위선이 잘리지 않게 할 것.  
2. **모델 연산 실수:** 삼각식에서 `cosθ`를 한 번 곱한 뒤 **또** 곱해 식이 틀어지는 패턴 등. 프롬프트에 **단계당 한 번의 정당한 변형**·**원방정식 대입 검산**을 명시함(`prompts.ts`, `generate-explanation/route.ts`).  
3. **메타 문장·자기모순:** “문제 오류” 등 → 서버 검증 `validateNoMetaUndermining`으로 재시도 유도.  
4. **문항 4번 빠른정답 비어 있음:** 카드의 **빠른정답** 필드가 비어 있으면 내보내기 점검에서 걸림. 생성 결과 파싱 실패 또는 수동 수정 누락일 수 있음 → 해당 문항 카드에서 빠른정답 입력.

## 4. 한계 (전문가 합의)

- LLM은 **100% 정답을 보장하지 않습니다.** 프롬프트·검증·교차검증은 **확률을 낮출 뿐**이며, 시험지·수능급은 **인간 검수**가 최종 방어선입니다.
